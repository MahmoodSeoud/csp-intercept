# Design: CSP link debugger — a gdb for transfers under loss

Generated by /office-hours on 2026-07-23
Branch: master · Repo: csp-intercept · Status: DRAFT · Mode: Builder
Supersedes: docs/cspx-assurance-tool-spec.md (that spec was verdict-centric and
standalone-Python; this reframes the tool as a debugger APM. See "What changed".)

## Problem

The operator works inside csh. When a transfer or a procedure fails under a lossy
link, there is no way to watch and dissect what is happening on the wire, live, in
the session, and reach your own conclusion. Today there is a passive sniffer that
writes a CSV only its author can read, and a loss injector with twelve positional
args. The concrete case: the DISCO-2 payload-access procedure spawns a csh session
on the payload; it works on the flatsat but fails on orbit because a command packet
is silently dropped, and nobody can see which one.

## What makes this the right tool

It is gdb for CSP. Not a black box that declares "CORRUPT" — an instrument that
shows ground truth (every frame decoded, exactly what the degrader dropped, the
state you can inspect on demand) and lets the operator conclude. For a thesis and
an assurance context this is the credible posture: show "reported DELIVERED, hash
MISMATCH, packet 147 dropped" and let the operator conclude "silent corruption,"
instead of an automated label that is wrong once and loses trust forever.

## Constraints

- Lives in csh as an APM (`apm load`), like `csp_monitor` and the SVU.
- `csp_monitor`'s promiscuous sniffer is **single-consumer** (its header forbids a
  second concurrent promisc reader). So the live view must live INSIDE the monitor,
  not a concurrent sibling APM.
- Flatsat-only. The payload-access procedure is root injection: keep the existing
  safety rules (health-gate, reset, guard), and the sensitive mechanism stays
  untracked.
- 4800 bps real uplink; pacing is already handled by existing tooling.
- Tuesday: usable to debug the payload-access failure live in front of Julian.

## Premises (agreed in this session)

1. APM in csh is the primary live surface.
2. Live packet dissection is first-class.
3. **The tool exposes state and control; the operator forms the conclusion. No
   authoritative verdict.** (revised from an earlier "the verdict is the headline")
4. Agent / sweep / report is a thin wrapper around csh from outside, not more APM.

## Approaches considered

Depth: **inspection-first (chosen)** vs full interactive debugger (breakpoints +
step) vs inspection-first-but-designed-for-full.

Architecture: **extend `csp_monitor` (chosen)** vs new sibling APM `cspdbg` vs
discuss.

### Approach A — Extend csp_monitor, inspection-first (CHOSEN)
Add a live dissect-print mode to the monitor APM: decode each CSP frame to the
console live with a port/node filter, reusing `ci_meas` and the existing promisc
hook. Inspection on demand uses existing csh commands. No verdict. One APM, one
sniffer. Effort S · Risk Low · reuses all of `csp_monitor` + `ci_meas`.

### Approach B — Sibling debugger APM `cspdbg`
Rejected for Tuesday: collides with the single-consumer sniffer rule, needs a
promisc-bus refactor first, more code and risk.

## Recommended approach

Approach A. gdb-style verbs mapped onto CSP:

| gdb | this tool (in csh) | built from |
|---|---|---|
| `run` | run the procedure through the degraded link | existing driver script |
| `watch` / `disassemble` | **live dissect view**: decode src/dst/port, RDP/DTP/param, flags, seq/offset; mark frames the degrader dropped | NEW, in the monitor |
| `print` / `x` | read back a param, hash a file, ping a node | existing csh `get` / `verify` / `ping` |
| `backtrace` | the op sequence and where the drop landed | the dissect trace |
| **the conclusion** | **the operator's, not the tool's** | — |

### The one genuinely new piece: the live view

Extend `csp_monitor` with a live, human-readable dissect stream. The architecture,
after eng-review, decouples capture from display:

```
  bus ──promisc──▶ drainer thread ──decode(ci_meas)──▶ decoded-line RING ──▶ foreground `dissect` ──▶ console
      (single consumer, never blocks)   drop-oldest, non-blocking        (main thread, tail -f)
                                                              ▲
  degrader drop.csv ──────────────────────────────────────────┘  (merged in: what was DROPPED)
```

- **Display path (eng-review D2 = A):** the drainer stays the sole promisc consumer
  and appends decoded lines to an **in-memory ring**; a **foreground** `dissect`
  (or `tail`) subcommand drains the ring to the console on the main thread, like
  `tail -f`, with a `-d <port>` / `-n <node>` filter. This kills BOTH the csh-prompt
  interleaving (no background printing) and the single-consumer conflict (no second
  sniffer reader). Do NOT print from the drainer thread.
- **Generic-frame branch (eng-review A1, essential):** the drainer today writes a
  row only for RDP (`csp_monitor_apm.c:106`) or DTP data (`:125`); every other
  frame, including the **param traffic** the payload-access procedure uses, falls
  through both branches and produces nothing. Add an else branch that emits
  src/dst/dport/flags/length for ANY frame, or the dissect view is blank for the
  Tuesday case.
- **Drop marking (eng-review A3, correction):** a dropped frame never reaches the
  bus, so the sniffer can't see it. The monitor infers drops only for RDP (seq
  gaps); param traffic has no sequence. So for the payload-access case the "what
  was dropped" stream comes ENTIRELY from the **degrader's `drop.csv`**, merged
  into the view by time. The monitor shows what crossed; the degrader log shows
  what didn't.
- **Ring is drop-oldest and non-blocking on the producer side (eng-review P1):** the
  drainer is the single promisc consumer and must never block, or it drops real
  packets and corrupts the measurement (the monitor already guards against this
  "instrument loss"). The display side may starve; the capture side never does.
  Single-producer / single-consumer, so a lock-free SPSC ring or a tiny mutex the
  drainer never waits on.

## What changed (and the fate of the Python verdict engine)

The earlier spec built a standalone Python `cspx` that post-processed capture CSVs
and emitted a PASS / FAIL_LOUD / FALSE_COMPLETE verdict. That is demoted. The
classification is NOT the product. At most it survives as an opt-in "facts summary"
in the OUTSIDE wrapper (`claimed=DELIVERED, sha=MISMATCH, 147 dropped`) with no
label attached. See open questions.

## Open questions

- Verdict logic: keep the built engine as an opt-in, non-authoritative facts
  summary in the outside wrapper, or delete it? (leaning: be careful with verdicts)
- Live-print vs the csh prompt: line-buffered stream + pause toggle; confirm it
  reads cleanly during a run.
- Param decoding: the monitor decodes RDP/DTP today; the payload-access traffic is
  the param service (generic frames). Tuesday can live with "generic frame to port
  X dropped"; a light param decode would make the WHERE more legible later.
- Naming: keep `csp_monitor` + a `dissect` subcommand, or brand the debug surface.

## Success criteria (Tuesday)

Live in csh, no CSV post-processing and no verdict label: load the monitor, start
the dissect view, run the payload-access procedure through the degrader at a loss
level, WATCH the command frame drop in the live trace, `get` the param and see it
still holds the prior value, and conclude out loud: "the write never landed, that
is why no session came up."
